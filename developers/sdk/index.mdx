# Plugin SDK

The plugin SDK is the stable API surface for WednesdayAI extensions. Plugins import from `openclaw/plugin-sdk` ã never from core internals or relative paths into `src/`.

## Installation

Add `openclaw` as a dev/peer dependency. Use `wednesdayai` (the primary package name) or `openclaw` (the backward-compatible alias ã both resolve to the same package):

```json
{
  "type": "module",
  "peerDependencies": {
    "openclaw": "*"
  },
  "devDependencies": {
    "openclaw": "^0.4.0"
  }
}
```

Never add `openclaw` to `dependencies` ã the gateway provides it at runtime.

## The plugin entry

A plugin module exports a default `register` function (sync or async) that receives the `OpenClawPluginApi`:

```ts
import type { OpenClawPluginApi } from "openclaw/plugin-sdk";

export default function register(api: OpenClawPluginApi): void {
  // register tools, hooks, channels, routes, etc.
}
```

A plugin may instead export an `OpenClawPluginDefinition` object with a `register` (or `activate`) method and optional `id` / `name` / `configSchema` / `ownsCompaction` fields. Both forms are accepted.

## The `api` object

`OpenClawPluginApi` is what your `register` function receives. Key fields and methods:

| Member | Purpose |
| ------ | ------- |
| `api.id`, `api.name`, `api.version` | Plugin identity |
| `api.config` | The resolved `OpenClawConfig` |
| `api.pluginConfig` | This plugin's config (`Record<string, unknown> \| undefined`) |
| `api.logger` | `PluginLogger` (`info` / `warn` / `error` / `debug?`) |
| `api.runtime` | Runtime sub-APIs (see below) |
| `api.registerTool(tool, opts?)` | Register an agent tool ([guide](/developers/agent-tools)) |
| `api.on(name, handler, opts?)` | Register a lifecycle hook ([guide](/developers/hooks)) |
| `api.registerHook(events, handler, opts?)` | Lower-level hook registration |
| `api.registerChannel(reg)` | Register a channel ([guide](/developers/channel-adapters)) |
| `api.registerHttpRoute(params)` | Register a gateway HTTP route |
| `api.registerGatewayMethod(method, handler)` | Register a gateway RPC method |
| `api.registerCli(registrar, opts?)` | Add CLI commands |
| `api.registerCommand(command)` | Register a chat command that bypasses the LLM |
| `api.registerService(service)` | Register a start/stop background service |
| `api.registerProvider(provider)` | Register a model provider |
| `api.registerWebSearchProvider(provider)` | Register a web-search provider |
| `api.registerRealtimeVoiceProvider(definition)` | Register a real-time voice provider (`RealtimeVoiceProviderDefinition`) |
| `api.registerWebhookTargetWithPluginRoute(path, handler)` | Register a webhook target scoped to the plugin's route namespace |
| `api.resolvePath(input)` | Resolve a path relative to the plugin |

## `api.runtime` sub-APIs

`api.runtime` exposes vetted runtime capabilities so plugins never reach into core internals:

| Sub-API | Key methods | Notes |
| ------- | ----------- | ----- |
| `runtime.identity` | `.name`, `.avatarUrl?`, `.description?`, `.locale?` | Agent identity / persona metadata |
| `runtime.api` | `.source: "channel" \| "webhook" \| "rest" \| "internal"`, `.description?: string` | Metadata about the originating API request |
| `runtime.analysis` | `run(params: AnalysisRunParams): Promise<AnalysisResult>`, `enqueue(params: AnalysisRunParams, onComplete?: (result: AnalysisResult) => void): AnalysisJob` | Plugin-owned focused LLM analysis. See [Analysis runtime](/developers/analysis-runtime). |
| `runtime.signals` | `publish(signal)`, `subscribe(filter, handler)`, `getAvailability()`, `requestWake()`, `nudge(agentId, message)` | Agent Signals bus. See [Agent signals](/developers/agent-signals). |
| `runtime.state` | `resolveStateDir(agentId?)` | Resolve the plugin/agent on-disk state directory |
| `runtime.config` | `loadConfig()`, `writeConfigFile(patch)` | Read or write the gateway config at runtime |
| `runtime.media` | `loadWebMedia(url)`, `detectMime(buf)`, `getImageMetadata(buf)`, `resizeToJpeg(buf, opts)`, `mediaKindFromMime(mime)`, `isVoiceCompatibleAudio(mime)` | Media fetch, detection, and transform utilities |
| `runtime.tts` | `textToSpeechTelephony(text, opts)` | Text-to-speech for voice/telephony channels |
| `runtime.stt` | `transcribeAudioFile(path, opts)` | Audio file transcription |
| `runtime.tools` | `createMemoryGetTool()`, `createMemorySearchTool()`, `registerMemoryCli(api)` | Built-in memory tool factories |
| `runtime.events` | `onAgentEvent(handler)`, `onSessionTranscriptUpdate(handler)` | Subscribe to agent lifecycle and transcript events |
| `runtime.system` | `enqueueSystemEvent(event)`, `requestHeartbeatNow()`, `runCommandWithTimeout(cmd, opts)`, `formatNativeDependencyHint(dep)` | System-level actions |
| `runtime.logging` | `getChildLogger(bindings?: Record<string, unknown>, opts?: { level?: LogLevel })`, `shouldLogVerbose()` | Scoped logger and verbosity check |
| `runtime.channel` | Channel-level utilities (chunking, routing, pairing, media, sessions, reactions, groups) | Internal channel runtime ã use only when building channel adapters |

<Warning>
  `resolveStateDir` is **no longer exported** as a top-level SDK symbol. Resolve the state
  directory via `api.runtime.state.resolveStateDir()` instead.
</Warning>

## Tool parameter schemas (typebox)

Tool `parameters` use `@sinclair/typebox`. Pin it to exactly `0.34.48`:

```json
{
  "dependencies": {
    "@sinclair/typebox": "0.34.48"
  }
}
```

Schema constraints:

- No `Type.Union`, `anyOf`, `oneOf`, or `allOf`.
- Use `Type.Optional(...)` instead of `... | null`.
- Use `stringEnum` / `optionalStringEnum` (exported from the SDK) for string enumerations.
- Never use `format` as a raw property name.

```ts
import { Type } from "openclaw/plugin-sdk";
import { stringEnum, optionalStringEnum } from "openclaw/plugin-sdk";

const parameters = Type.Object({
  mode: stringEnum(["fast", "thorough"]),
  region: optionalStringEnum(["us", "eu"]),
});
```

## Commonly used exports

These are exported from `openclaw/plugin-sdk` and are safe to depend on:

- **Types** ã `OpenClawPluginApi`, `OpenClawPluginDefinition`, `AnyAgentTool`, `ChannelPlugin` (and the `Channel*Adapter` family), `PluginRuntime`, `OpenClawConfig`, the `Analysis*` types, the `AgentSignal*` types, `AgentAvailability`, `RealtimeVoiceProviderDefinition`, `VoiceSession`, `ProviderPlugin`, `ProviderCapabilities`, `GatewayRequestHandler`, `GatewayRequest`, `GatewayResponse`, `WebSearchProvider`, `WebSearchOptions`, `WebSearchResult`.
- **Hook payload types** ã `HookPayload`, `MessageReceivedPayload`, `MessageSentPayload`, `SessionStartPayload`, `SubagentSpawningPayload`, `BeforeFileWritePayload`.
- **Tool helpers** ã `jsonResult`, `readStringParam`, `readNumberParam`, `readReactionParams`, `createActionGate`, `ToolAuthorizationError`.
- **Schema helpers** ã `stringEnum`, `optionalStringEnum`, `emptyPluginConfigSchema`.
- **Logging** ã `createSubsystemLogger`, `getChildLogger`, `FileLog`.
- **Config/secrets** ã `loadConfig`, `isSecretRef`, `SecretInput` / `SecretRef` types.
- **HTTP/webhooks** ã `registerPluginHttpRoute`, `registerWebhookTarget`, request-guard helpers.

<Note>
  `ToolDefinition`, `inputSchema`, `run()`, `HookContext`, and `PluginDefinition.tools[]`
  do **not** exist in this SDK. Tools are `AnyAgentTool` objects with `parameters` and
  `execute(_id, params)`, registered via `api.registerTool(...)`. Hooks are registered via
  `api.on(...)`.
</Note>

## Manifest vs package.json key

Two related-but-distinct things both use the word "openclaw":

- **`openclaw.plugin.json`** ã the manifest at the plugin root. Carries `id` and `configSchema` and is used to validate config **without executing plugin code**. Required for every plugin. See the [manifest reference](/developers/plugins/manifest).
- **`package.json` `"openclaw"` key** ã used for workspace/pack discovery (for example the `id`, or `extensions` for a multi-plugin pack).

A typical published plugin ships both: the `package.json` `"openclaw"` key for discovery, and `openclaw.plugin.json` for config validation.

## Extending the SDK surface

When you need a type or contract that is not yet exported:

1. Do not import it from core internals ã that creates a hard coupling that breaks on upgrade.
2. Open an issue or PR to add it to `src/plugin-sdk/` with documentation.
3. Keep exports typed. No `any` in the SDK surface.

See [Contributing](/developers/contributing) for the workflow.
